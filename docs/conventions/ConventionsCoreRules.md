# 🛠️ CORE_RULES (전체 프로젝트 공통 개발 지침)

이 문서는 웹과 앱 모든 프로젝트 소스 코드에 무조건적으로 적용되는 공통 개발 원칙이다. AI 에이전트는 코드를 생성하거나 리팩토링할 때 본 지침을 절대 준수하여 결과물의 품질을 유지한다.

---

## 🧱 1. 코드 구조 및 분리 규칙 (300줄 제한)

* **단일 파일 300줄 상한선:** 파일 하나당 **최대 300줄**을 절대 넘기지 않는다.
* **선제적 모듈 분리:** 구현 또는 리팩토링 과정에서 파일이 300줄을 초과할 가능성이 포착되면, 작업을 즉시 중단하고 모듈 분리 및 구조 분해를 먼저 수행한다.
* **기능 단위 독립성:** 모든 파일은 단일 책임 원칙(Single Responsibility Principle)에 따라 명확한 기능 단위로 쪼개어 분리함으로써 유지보수성을 극대화한다.
* **공통 컴포넌트화:** 2회 이상 반복되거나 재사용 가능성이 있는 코드는 무조건 독자적인 컴포넌트, 커스텀 훅, 또는 유틸리티 모듈로 분리한다.

---

## 🧹 2. 코드 품질 및 린트(Lint) 규칙 (동적 위임 구조)

* **[초엄격 린트 문서 연동]:** AI 에이전트는 코드를 작성하거나 수정하기 전, 반드시 `/docs/conventions/LintRules.md`에 명시된 상세 스타일 가이드를 완벽히 파싱하고 준수해야 한다.
* **[린트 스크립트 강제 완수]:** 모든 코드 생성 및 변경 작업이 완료되면 사용자에게 보고하기 전, 프로젝트 내 설정된 ESLint/Prettier 검증 명령어를 무조건 직접 실행하여 스타일을 정렬하고 에러 및 경고를 스스로 100% 해결해야 한다.
* **미사용 자원 즉시 제거:** 선언 후 사용하지 않는 변수, 임포트(Import), 함수는 코드에 남겨두지 말고 발견 즉시 제거한다.
* **리터럴 상수화:** 코드 내에 작성된 매직 넘버나 문자열 등 반복 사용되는 리터럴 값은 무조건 별도의 상수 파일(`constants`)이나 파일 상단에 명시적으로 상수화하여 관리한다.

## 🧪 3. 테스트 및 개발 문서 자동 동기화 규칙 (3-Way Synchronization)

한 가지 기능이 새로 추가되거나 기존 로직이 수정되면 소스 코드 구현뿐만 아니라, 테스트 케이스와 프로젝트 문서가 항상 삼위일체로 최신화 상태를 유지하도록 다음 절차를 무조건 수행한다.

* **[테스트 코드 동기화] 선(先) 탐색 및 업데이트:**
  * 코드를 수정하기 전, 해당 기능과 매핑되는 테스트 케이스 파일을 먼저 탐색한다.
  * **[케이스가 없는 경우]:** 정상 흐름 및 예외 분기를 검증하는 테스트 케이스를 새롭게 **추가**한다.
  * **[케이스가 있는 경우]:** 변경되거나 개선된 기능 스펙에 맞춰 기존 테스트 코드를 **업데이트**한다.
  * 작업 완료를 사용자에게 보고하기 전에 반드시 테스트 코드를 실행하며, 모든 테스트가 100% 통과한 경우에만 진행한다. 테스트 실패 시 사용자에게 보고하지 말고 먼저 스스로 수정한다.
* **[개발 문서 동기화] `/docs` 하위 최신화:**
  * 코드 구현 및 테스트 통과가 완료되면, `/docs` 디렉터리 하위의 관련 개발 문서(기능 정의서, 아키텍처 가이드 등)를 전체 탐색한다.
  * **[내용이 없는 경우]:** 새로 추가된 기능의 비즈니스 요구사항, 데이터 규칙, 엔지니어링 구조 명세를 관련 문서에 새롭게 **추가**한다.
  * **[내용이 있는 경우]:** 변경된 구현 방식이나 API 명세에 맞춰 기존 문서의 내용을 최신 상태로 **업데이트**한다.

## 🛡️ 4. 예외 분기 처리 및 사이드 이펙트 방어 규칙 (UI/상태 파괴 방지)

* **[필수] 완결형 분기 처리 (Defensive Branching):** 미디어 쿼리, 해상도 대응, 가로/세로 모드 변환 등의 조건문을 작성할 때, 특정 케이스만 고려하는 불완전한 분기를 금지한다. `if`문이나 삼항 연산자는 예측 가능한 모든 케이스를 커버해야 하며, 반드시 `else`나 기본값(Fallback)을 명시하여 예기치 못한 환경에서도 레이아웃이 파괴되지 않도록 방어한다.
* **[구조] 독립적 상태 격리 (State Isolation):** 특정 화면이나 모드의 기능을 개선 또는 수정할 때, 해당 영역의 상태(State)와 스타일이 다른 독립된 기능이나 디바이스 환경에 연쇄적인 사이드 이펙트를 일으키지 않도록 구조를 엄격히 분리한다.
