# 🌐 WEB_CONVENTIONS (웹 특화 규칙)

이 문서는 `Next.js` 및 웹 프론트엔드/백엔드 아키텍처 환경에만 특화되어 적용되는 개발 컨벤션이다.

---

## 📁 1. 디렉터리 구조 및 라우팅 규칙
* **Next.js App Router 아키텍처 준수:** 모든 페이지와 API 라우트는 App Router 컨벤션(`src/app/...`)을 철저히 따른다.
* **기능 중심 컴포넌트 분리:** 웹 전용 공통 컴포넌트는 `components/web` 하위에 격리하며, 특정 페이지에서만 공유되는 컴포넌트는 해당 라우트 폴더 내부에 독립적으로 위치시킨다.

---

## 🧠 2. 전역 상태 및 데이터 관리
* **Zustand 최적화 활용:** 전역 상태 관리는 `Zustand`를 사용하되, 무분별한 전역 스토어 확장을 지양하고 도메인/기능 단위로 명확히 분리하여 설계한다.
* **불필요한 리렌더링 방지:** 컴포넌트가 스토어의 전체 상태를 구독하지 않도록, 필요한 슬라이스(Slice)나 셀렉터(Selector) 형태로 쪼개어 가져오도록 코드를 통제한다.

---

## 🔒 3. 보안 및 인프라 최적화 (하루상자 기반)
* **악성 매크로 및 스코어 조작 방어:** 외부 라이브 서버에 노출되는 주요 API(예: 게임 점수 제출, 통계 기록 등)는 단순 IP 기반 레이트 리밋에 의존하지 않는다. 클라이언트 플레이 검증 단계를 통과해야만 서버가 유효한 토큰을 발급하는 **챌린지 토큰 인증 아키텍처** 구조를 반드시 반영한다.
* **독립 인프라 및 데이터 다이어트:** Nginx 리버스 프록시 및 PM2 무중단 운영 환경을 고려하여 에러 핸들링을 철저히 하고, 서버 성능 저하 및 인프라 비용을 발생시키는 불필요한 데이터 축적을 막기 위해 MongoDB/Mongoose 설계 시 데이터 만료 시간(TTL) 및 다이어트 설정을 반영한다.
* **SEO 최적화:** 구글 SEO 최적화를 위해 시맨틱 태그 구조를 철저히 지키고, 메타데이터 규칙을 페이지 컴포넌트마다 완벽하게 반영한다.
