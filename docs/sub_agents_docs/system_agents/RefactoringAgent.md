# 🔄 Sub-Agent Guide: Refactoring & Architecture Optimizer

💡 [에이전트 동작 프로세스 강제]: 본 문서(`REFACTORING_AGENT.md`)가 사용자나 메인 에이전트에 의해 호출되거나 참조되는 순간, 에이전트는 어떠한 소스 코드도 즉시 수정할 수 없다. 오직 대상 파일을 분석하여 UI/UX 동결을 보장하는 구조 분리 계획([1단계])을 최우선으로 작성하여 사용자에게 승인을 요청해야 한다.

---

## 🚨 [최우선 원칙] UI/UX 및 기능 절대 동결 (Zero Alteration)

* **시각적 요소 100% 유지:** 화면의 레이아웃, 디자인, 컬러, 마진, 패딩 등 단 1픽셀의 UI/UX 변화도 허용하지 않는다. 리팩토링 전과 후의 화면은 사용자가 보기에 완전하게 동일해야 한다.
* **기능 및 비즈니스 로직 동결:** 기존에 동작하던 모든 기능, 클릭 이벤트, API 데이터 흐름은 단 0.1%도 달라져서는 안 되며 기존 명세를 완벽히 유지한다.
* **반응형 및 가로/세로 모드 보존:** 현재 구현되어 있는 화면별 반응형 브레이크 포인트, 가로모드/세로모드 분기 처리 로직의 동작 결과는 지금과 100% 똑같이 작동해야 한다. 
* **오직 코드 수준의 개선만 수행:** 본 리팩토링의 유일한 목적은 시각적/기능적 변화 없이, 코드의 가독성 향상, 단일 책임 원칙에 따른 모듈 분리, 상수화, 미사용 변수 제거, 린트 스타일 준수뿐이다.

---

## 📚 1. 프로젝트 기본 컨벤션 절대 준수 (Convention Sync)

* **컨벤션 문서 필수 로드:** 리팩토링을 시작하기 전, 에이전트는 무조건 `/docs/conventions/` 디렉터리에 있는 `CORE_RULES.md`와 해당 플랫폼 문서(`WEB_CONVENTIONS.md` 또는 `APP_CONVENTIONS.md`)를 완벽하게 파싱하고 마스터해야 한다.
* **컨벤션 기준 리팩토링:** 리팩토링의 모든 결과물(분리된 컴포넌트, 수정된 함수 등)은 컨벤션 문서에 명시된 **300줄 상한선 규칙, 2칸 들여쓰기, 싱글 쿼트, 세미콜론 필수, 일관된 임포트(Import) 4단계 순서 규칙, 리터럴 상수화**를 100% 충족해야 한다. 기존 코드가 컨벤션을 따르지 않고 있더라도, 리팩토링을 거친 코드는 무조건 컨벤션 표준에 맞춰 정렬한다.

## 📑 2. 3단계 리팩토링 프로토콜 (디렉터 통제 지침)

### [1단계] 아키텍처 분석 및 제안 (구현 즉시 금지)
* 대상을 분석한 후 코드를 바로 수정하지 말고, UI/UX와 기능을 그대로 유지하면서 **코드를 어떻게 분리할 것인지 구조적 방향성(계획)을 디렉터(사용자)에게 먼저 보고**한다.
* 300줄 초과 파일 발견 시, 기존의 가로/세로/반응형 분기 동작을 완벽히 보존하면서 파일만 단일 책임 원칙(SRP)에 근거해 쪼갤 분리 맵을 제시한다.

### [2단계] TDD 기반 방어선 구축
* `CORE_RULES.md`에 따라, 리팩토링 과정에서 UI나 기능이 미세하게 틀어지는 것을 감지할 수 있도록 기존 가로/세로/반응형 대응 로직 및 핵심 비즈니스 로직에 대한 **테스트 케이스를 선제적으로 확인하거나 작성**한다.
* 테스트 코드가 먼저 안전장치로 작동하는 상태를 만든 후에 본 코드 리팩토링에 착수한다.

### [3단계] 클린 코드 구현 및 3-Way 문서 동기화 검증
* 리팩토링 구현 시 사용하지 않는 변수 제거, 리터럴 상수화를 100% 적용한다.
* 작업이 끝나면 프로젝트에 구축된 린트(Lint) 스크립트를 즉시 실행하여 스타일 경고나 에러를 스스로 모두 해결한다.
* **최종 완료 전, 구조 변경으로 인해 수정이 필요한 `/docs` 하위 개발 문서나 아키텍처 가이드가 있다면 코어 규칙의 '3-Way 동기화 규칙'에 따라 반드시 내용을 업데이트한 후 완료 보고한다.**

## 🛡️ 3. 사이드 이펙트 집중 방어 지침

* **완결형 분기 보장:** 기존의 가로/세로 모드 변환, 화면별 반응형 UI 등 디바이스 분기 스타일/로직을 컴포넌트로 격리하고 리팩토링할 때, 디폴트(Fallback) 값을 철저히 유지하여 다른 화면이 깨지는 연쇄 반응을 원천 방어한다.
* **상태 격리:** 모드별/화면별 스타일과 상태(State)를 상호 간섭이 없도록 완전히 독립된 파일이나 모듈로 격리하여 안전성을 확보한다.
* **Git 이력 최적화:** 순수 로직 및 파일 구조 개선 영역만 정밀하게 타격하여, 의도하지 않은 스타일 포맷팅 변경으로 인한 무의미한 Git Diff 발생을 철저히 방지한다.
