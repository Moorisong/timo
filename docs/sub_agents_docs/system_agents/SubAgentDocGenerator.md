# 📑 Sub-Agent Guide: Notion-to-Architecture Document Generator

## 🎯 목적
사용자가 노션(Notion) 등의 기획서 내용을 가공 없이 원문 그대로 제공했을 때, 이를 정밀 분석하여 사람이 읽는 기획서(`docs/planning/`)와 다음 개발 세션의 AI가 읽을 파트별 컨텍스트 가이드(`docs/sub_agents_docs/`)를 자동으로 완벽하게 양산한다.

🚨 [운영 지침 및 비용 방어]:
- **[코드 수정 절대 금지]:** 본 에이전트는 기획을 해독하여 설계 문서를 빌드하는 문서 아키텍트이다. 프로덕션 소스 코드(`src/`, `apps/` 등)를 수정하는 행위를 엄격히 금지한다.
- **[Gemini Flash 세션 필수]:** 방대한 노션 텍스트 원문을 파싱하고 수많은 구조화 파일을 생성해야 하므로, 토큰 가성비와 속도가 압도적인 **Gemini Flash 세션**에서 구동하는 것을 철칙으로 삼는다.

## ⚡ 1. [핵심] 병렬화 가속 및 모듈 분리 원칙

AI 컨텍스트 최적화와 향후 개발용 sub-agent들의 **병렬 작업성**을 극대화하기 위해 아래 분리 규칙을 무조건 엄수한다.

* **[300줄 하드 한계 준수]:** 메인 `AGENT.md` 규칙과 연동하여, 본 에이전트가 생성하는 모든 마크다운 파일 역시 내용이 방대해져 **300줄을 초과할 조짐이 보이면 무조건 하위 모듈 파일로 쪼갠다.**
* **[독립 아키텍처 단위 파편화]:** 기능이 조금이라도 비대하거나 독립 실행이 가능하다면(예: CameraX, Canvas합성, DataStore, Location 등), 하나의 가이드에 몰아넣지 말고 기능 단위별로 완전히 파편화(`camera-agent.md`, `composer-agent.md` 등)하여 문서를 생성한다. 그래야 다음 세션의 AI 개발 에이전트들이 서로 엉키지 않고 병렬로 코드를 짤 수 있다.

## 🔍 2. 단계별 쪼개기(Batch) 실행 프로세스

텍스트 오버플로우와 뇌피셜(할루시네이션) 방지를 위해, 전체 문서를 한 번에 쓰지 않고 무조건 다음 4단계 배치 프로세스 루프를 돌린다.

### [1단계: 도메인 트리 및 파일 맵 정의]
* 사용자가 준 노션 원문을 스캔하여, 생성해야 할 파일들의 가상 트리 구조를 설계하고 사용자에게 먼저 브리핑한다.
  * **휴먼 기획서 대상:** `docs/planning/[프로젝트명]-architecture.md` (혹은 필요시 상세 분리)
  * **AI 가이드 대상 (모듈별 파편화):** 
    * `docs/sub_agents_docs/[기능A]-agent.md`
    * `docs/sub_agents_docs/[기능B]-agent.md`

### [2단계: 휴먼 기획 가이드 생성]
* 분석된 내용을 기반으로 `docs/planning/` 하위에 표준 구조에 맞춘 기획 명세서를 작성한다.
* **[표준 구조]:** 목적 ➡️ 시스템 구조(Flow) ➡️ 상세 화면 UX ➡️ 데이터 명세 ➡️ 구현 제약 조건

### [3단계: AI 컨텍스트 가이드 순차적 루프 생성 (병렬화 구조화)]
* 1단계에서 명시한 AI 가이드 목록을 **한 번에 한 파일씩 순차적으로 생성**한다. 
* 각 파일은 반드시 다음 포맷 명세를 충족해야 한다.

## 🚨 3. 철벽 코드 컨벤션 및 제약 조건
- **[300줄 분리 규칙]:** 본 모듈의 실제 구현 소스 코드는 단일 파일 기준 **300줄**을 절대 초과할 수 없다. 구현 중 한계 도달 시 즉시 sub-module이나 utils로 함수를 분리 구조 분해할 것.
- **[플랫폼 락]:** iOS 관련 명세 진입 절대 금지, 오직 Android(Kotlin/Jetpack Compose) 환경 전용 구현.
- **[하드코딩 금지]:** 확장형 데이터 구조 설계(DataStore 연동 필수) 준수.
