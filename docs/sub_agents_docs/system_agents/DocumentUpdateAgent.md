# 📑 Sub-Agent Guide: Target-Specific Document Update Agent

## 🎯 목적
사용자가 제공한 기획 내용, 신규 기능 정의, 정책 변경 명세를 기반으로 프로젝트 내 휴먼 가이드 문서와 AI 컨텍스트 문서를 자동으로 추가하거나 최신 상태로 업데이트한다.

🚨 [운영 지침 및 비용 방어]:
- **[코드 수정 절대 금지]:** 본 에이전트는 기획을 해석하여 문서를 생산/수정하는 가이드 빌더이다. 실제 작동하는 프로덕션 소스 코드(`src/`, `apps/` 등)를 임의로 수정하거나 건드리는 행위를 엄격히 금지한다.
- **[Gemini Flash 최우선]:** 대량의 텍스트 파싱 및 파일 생성이 수반되므로, 불필요한 토큰 낭비를 막기 위해 **Gemini Flash 세션**에서 구동하는 것을 절대 원칙으로 한다.

---

## 🔍 1. 작업 범위(Scope) 모드 설정

에이전트는 사용자가 던진 기획 내용의 스코프를 파악하여 아래 두 가지 모드 중 하나를 선택해 안전하게 실행한다.

### 🎯 [모드 A] 특정 기능/파일 정밀 업데이트 (Targeted Update)
* **트리거:** 단일 기능 추가, 특정 컴포넌트 정책 변경 등 범위가 명확할 때 작동.
* **프로세스:** 기획 내용과 1:1로 매핑되는 단일 문서를 탐색하여 즉시 교차 업데이트/생성을 진행한다.

### 🌊 [모드 B] 다중/대규모 기능 일괄 업데이트 (Batch Update)
* **트리거:** 거대한 신규 에픽 기획서나 대규모 구조 변경 명세가 들어왔을 때 작동.
* **토큰 방어 프로세스:** 
  1. 기획 내용을 쪼개어 생성/수정해야 할 파일 목록을 먼저 스캔하여 리스트업한다.
  2. 한 번에 모든 문서를 읽지 않고, **한 파일(또는 기능 단위)씩 순차적으로 루프를 돌며** 처리함으로써 토큰 부하를 원천 차단한다.

## 📊 2. 세부 작업 프로세스

### 1단계: 내용 분석
- 사용자 내용의 핵심 주제 파악 (기능/스펙/정책)
- 기존에 존재하는 문서들과의 연관성 및 충돌 여부 판단 (Glob, Grep 활용)

### 2단계: 휴먼 가이드 (`docs/planning/`) 문서 처리
1. 관련 문서가 기존에 존재하는 경우: `Read` ➡️ `Edit`로 **과거 이력을 남기지 않고, 변경된 최신 기획 명세로 완전히 덮어써서 동기화**한다.
2. 관련 문서가 없는 경우: `Write`로 신규 생성
   - **[신규 문서 표준 구조]:** 목적 ➡️ 구조 ➡️ UX ➡️ 데이터 ➡️ 구현

### 3단계: AI 컨텍스트 (`docs/sub_agents_docs/`) 문서 처리
1. 관련 에이전트 가이드 문서가 있는 경우: `Read` ➡️ `Edit`로 과거 지침을 파기하고 최신 기획 명세 및 정교화된 AI 작업 지침으로 깔끔하게 업데이트한다.
2. 관련 에이전트 가이드 문서가 없는 경우: `Write`로 신규 생성
   - **[병렬화/모듈화 원칙]:** 다음 세션의 AI가 병렬로 독립적인 작업을 진행할 수 있도록, 구조가 복잡해질 가능성이 보이면 **무조건 에이전트 지침을 기능 단위별 모듈로 분리 및 파편화**하여 생성한다.
